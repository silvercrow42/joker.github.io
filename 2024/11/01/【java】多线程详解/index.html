# 并发基础

# 什么是并发

并发是指多个任务在同一时间段内交错执行的能力。虽然这些任务可能不是真正的同时执行，但它们在时间上被快速交替调度，从而给人一种同时进行的错觉。并发可以显著提高程序的效率和响应性。以下是并发的几个关键点：

  • 任务切换：操作系统通过快速切换任务来实现并发，每个任务在一段时间内获得 CPU 时间片。
  • 资源共享：并发任务通常共享某些资源，如内存、文件句柄等，因此需要适当的同步机制来避免冲突。
  • 提高资源利用率：通过并发，可以更充分地利用系统资源，尤其是多核处理器。

# 并发的使用场景

  1. Web 服务器
    • 处理多个请求：Web 服务器需要同时处理来自多个客户端的请求。通过并发，服务器可以创建多个线程或进程来处理这些请求，提高响应速度和吞吐量。
    • 异步 IO 操作：处理网络请求和文件读写等 IO 操作时，使用异步 IO 可以避免阻塞主线程，提高整体性能。
  2. 数据库系统
    • 并行查询：数据库系统可以并行执行多个查询，特别是在处理大数据集时，可以显著提高查询效率。
    • 事务管理：并发控制机制（如锁和事务隔离级别）确保多个事务可以安全地并行执行，避免数据不一致。
  3. 图形用户界面（GUI）应用程序
    • 响应用户输入：主线程负责处理用户输入和更新界面，后台线程处理耗时任务（如数据加载、计算等），保持界面的响应性。
    • 动画和渲染：并行处理动画和渲染任务，提高视觉效果的流畅度。
  4. 大数据分析
    • 并行数据处理：使用 MapReduce 等并行计算框架，将大数据集分成多个部分，由多个节点并行处理，加快数据处理速度。
    • 分布式存储：并行读取和写入分布式存储系统中的数据，提高数据访问效率。
  5. 游戏开发
    • 物理模拟：并行处理复杂的物理模拟计算，提高游戏的真实感和性能。
    • AI 计算：并行处理游戏中的 AI 逻辑，使 NPC（非玩家角色）的行为更加智能和多样化。
  6. 文件和数据处理
    • 并行文件处理：并行读取和处理多个文件，加快数据处理速度。
    • 批处理任务：并行执行批处理任务，如日志分析、数据转换等。
  7. 网络爬虫
    • 并行抓取：同时从多个网站抓取数据，提高抓取速度。
    • 数据处理：并行处理抓取到的数据，加快数据清洗和分析过程。
  8. 机器学习和深度学习
    • 模型训练：并行训练模型，特别是在使用 GPU 加速时，可以显著缩短训练时间。
    • 数据预处理：并行处理大量训练数据，提高数据准备效率。
  9. 实时系统
    • 实时数据处理：并行处理实时数据流，如传感器数据、股票市场数据等，确保低延迟和高吞吐量。
    • 任务调度：并行调度和执行多个实时任务，确保系统的响应性和可靠性。
  10. 云计算和虚拟化
    • 资源分配：并行管理和分配计算资源，提高资源利用率。
    • 虚拟机管理：并行处理虚拟机的创建、启动和停止等操作，提高管理效率。

# 高并发情况下可能出现的问题

在并发场景下，可能会面临多种问题，这些问题如果不妥善处理，会影响系统的稳定性和性能。以下是一些常见的并发问题及其简要说明：

  1. 死锁（Deadlock）
    • 定义：多个线程或进程互相等待对方持有的资源，导致所有线程或进程都无法继续执行。
    • 原因：资源分配不当、循环等待、缺乏超时机制等。
    • 示例：线程 A 持有资源 X 并请求资源 Y，而线程 B 持有资源 Y 并请求资源 X。
  2. 竞态条件（Race Condition）
    • 定义：多个线程或进程以不可预测的顺序访问和修改共享资源，导致不一致的状态。
    • 原因：缺乏适当的同步机制、线程间通信不正确等。
    • 示例：多个线程同时读取和修改同一个变量，导致最终结果不确定。
  3. 上下文切换开销（Context Switching Overhead）
    • 定义：操作系统在不同线程或进程之间切换时，需要保存和恢复当前线程的上下文，这会消耗时间和资源。
    • 原因：频繁的上下文切换。
    • 示例：在高并发环境下，频繁的线程切换会导致 CPU 利用率下降。
  4. 资源竞争（Resource Contention）
    • 定义：多个线程或进程竞争有限的资源，导致性能下降或资源浪费。
    • 原因：资源不足、同步机制不当等。
    • 示例：多个线程同时访问同一个文件或数据库连接池。
  5. 饥饿（Starvation）
    • 定义：某个线程或进程由于长时间无法获得所需的资源而无法继续执行。
    • 原因：优先级调度不当、资源分配不公平等。
    • 示例：低优先级线程长时间无法获得 CPU 时间。
  6. 活锁（Livelock）
    • 定义：多个线程或进程不断重复相同的操作，试图避免冲突，但最终无法取得进展。
    • 原因：缺乏有效的冲突解决机制。
    • 示例：两个线程在尝试获取同一个资源时，不断退让，导致两者都无法成功获取资源。
  7. 内存泄漏（Memory Leak）
    • 定义：程序在运行过程中未能释放不再使用的内存，导致内存占用逐渐增加。
    • 原因：未正确管理内存分配和释放、对象引用未解除等。
    • 示例：线程创建的对象未被及时回收，导致内存占用不断增加。
  8. 不一致状态（Inconsistent State）
    • 定义：由于并发操作导致数据状态不一致，影响程序的正确性。
    • 原因：缺乏原子操作、同步机制不当等。
    • 示例：多个线程同时修改同一个数据结构，导致数据结构损坏。
  9. 性能瓶颈（Performance Bottleneck）
    • 定义：系统在高并发情况下，由于某个组件的性能限制，导致整体性能下降。
    • 原因：单点故障、资源限制、设计不合理等。
    • 示例：数据库连接池大小限制导致请求排队，影响整体性能。

解决方法

  • 同步机制：使用锁（如互斥锁、信号量）、条件变量等同步机制来协调线程之间的访问。
  • 原子操作：使用原子操作来确保操作的不可分割性。
  • 线程池：使用线程池来管理线程，减少上下文切换开销。
  • 资源管理：合理分配和管理资源，避免资源竞争和饥饿。
  • 超时机制：设置超时机制，防止死锁和活锁。
  • 内存管理：及时释放不再使用的内存，避免内存泄漏。
  • 性能优化：识别和优化性能瓶颈，提高系统整体性能。

# Happen-before 原则

Happen-before 原则是 Java 内存模型（Java Memory Model, JMM）中的一个重要概念，用于保证多线程环境下的内存可见性和操作顺序。Happen-before
原则定义了两个操作之间的顺序关系，确保一个操作的结果对另一个操作是可见的。

如果一个操作 A happen-before 另一个操作 B，那么 A 的结果对 B 是可见的，且 A 的动作在 B 之前完成。

Happen-before 原则包括以下几个具体规则：

  • 程序顺序规则（Program Order Rule）
    • 在一个线程中，按照程序代码的顺序，前面的操作 happen-before 后面的操作。
    • 示例：在同一个线程中，x = 1; y = x + 1;，x = 1 happen-before y = x + 1。
  • 监视器锁规则（Monitor Lock Rule）
    • 对同一个锁的解锁操作 happen-before 后面对该锁的加锁操作。
    • 示例：线程 A 解锁一个对象的监视器，然后线程 B 加锁同一个对象的监视器，A 的解锁操作 happen-before B 的加锁操作。
  • volatile 变量规则（Volatile Variable Rule）
    • 对一个 volatile 变量的写操作 happen-before 后面对该变量的读操作。
    • 示例：线程 A 写入一个 volatile 变量 x，然后线程 B 读取同一个 volatile 变量 x，A 的写操作 happen-before B 的读操作。
  • 线程启动规则（Thread Start Rule）
    • 线程的启动操作 happen-before 线程中的任何操作。
    • 示例：线程 A 启动线程 B，A 的启动操作 happen-before B 中的任何操作。
  • 线程终止规则（Thread Termination Rule）
    • 线程中的任何操作 happen-before 线程的终止检测。
    • 示例：线程 A 终止，线程 B 检测到 A 已经终止，A 中的任何操作 happen-before B 检测到 A 的终止。
  • 中断规则（Interrupt Rule）
    • 对线程的中断操作 happen-before 被中断线程检测到中断。
    • 示例：线程 A 中断线程 B，A 的中断操作 happen-before B 检测到中断。
  • 传递性（Transitivity）
    • 如果 A happen-before B，且 B happen-before C，则 A happen-before C。
    • 示例：A happen-before B，B happen-before C，因此 A happen-before C。

# Java 中的线程

# 线程的生命周期

Java 线程有六种状态，这些状态反映了线程在其生命周期中的不同阶段。

  1. 新建（New）：线程被创建但尚未启动。

    前置条件：

    • 当调用 Thread 对象的 start 方法时，线程从新建状态切换到就绪状态。

  2. 就绪（Runnable）线程已经准备好运行，等待 CPU 分配时间片。

    前置条件：

    • 从新建状态调用 start 方法后进入就绪状态。
    • 从阻塞、等待或定时等待状态返回后进入就绪状态。
    • 从运行状态因时间片用完或更高优先级线程抢占 CPU 而进入就绪状态。

  3. 运行（Running）：Java 线程中将就绪（ready）和运行中（running）两种状态笼统的称为 “运行”，以下为两种状态的区分情况：

    • 就绪（Ready）：线程已经准备好运行，等待 CPU 分配时间片。

      前置条件：

      • 从新建状态调用 start 方法后进入就绪状态。
      • 从阻塞、等待或定时等待状态返回后进入就绪状态。
      • 从运行状态因时间片用完或更高优先级线程抢占 CPU 而进入就绪状态。

    • 运行（Running）：线程正在 CPU 上执行。

      前置条件：

      • 从就绪状态获得 CPU 时间片后进入运行状态。
      • 从运行状态因时间片用完或更高优先级线程抢占 CPU 而进入就绪状态。
      • 从运行状态因调用 wait、join、sleep 等方法进入等待或定时等待状态。
      • 从运行状态因调用 Object 的 wait 方法进入等待状态。
      • 从运行状态因调用 Thread 的 join 方法进入等待状态。
      • 从运行状态因调用 Thread 的 sleep 方法进入定时等待状态。

  4. 阻塞（Blocked）：线程被阻塞，等待获取监视器锁。

    前置条件：

    • 从运行状态因尝试获取已被其他线程持有的锁而进入阻塞状态。
    • 从阻塞状态因获得锁后进入就绪状态。

  5. 等待（Waiting）：线程无限期等待其他线程执行特定操作。

    前置条件：

    • 从运行状态因调用 Object 的 wait 方法进入等待状态。
    • 从运行状态因调用 Thread 的 join 方法进入等待状态。
    • 从运行状态因调用 LockSupport 的 park 方法进入等待状态。
    • 从等待状态因调用 Object 的 notify 或 notifyAll 方法进入就绪状态。
    • 从等待状态因调用 Thread 的 interrupt 方法进入就绪状态。

  6. 定时等待（Timed Waiting）：线程在指定时间内等待其他线程执行特定操作。

    前置条件：

    • 从运行状态因调用 Thread 的 sleep 方法进入定时等待状态。
    • 从运行状态因调用 Object 的 wait 方法并指定超时时间进入定时等待状态。
    • 从运行状态因调用 Thread 的 join 方法并指定超时时间进入定时等待状态。
    • 从运行状态因调用 LockSupport 的 parkNanos 或 parkUntil 方法进入定时等待状态。
    • 从定时等待状态因超时或调用 Thread 的 interrupt 方法进入就绪状态。

  7. 终止（Terminated）：线程已经退出，不再活动。

    前置条件：

    • 从运行状态因 run 方法执行完毕或抛出未捕获的异常而进入终止状态。
    • 从等待或定时等待状态因调用 Thread 的 interrupt 方法并抛出 InterruptedException 而进入终止状态。

# 线程的创建

# 常规方式创建

  • 通过继承 Thread 类创建线程：

    1. 创建一个新的类，该类继承自 Thread 类。
    2. 重写 run 方法，在其中定义线程执行的代码。
    3. 创建该类的实例，并调用 start 方法启动线程。

  • 通过实现 Runnable 接口创建线程：

    1. 创建一个新的类，该类实现 Runnable 接口。
    2. 实现 run 方法，在其中定义线程执行的代码。
    3. 创建 Runnable 接口的实现类实例，并将其传递给 Thread 类的构造器。
    4. 调用 Thread 对象的 start 方法启动线程。

# 线程池

Java 中的线程池是一种用于管理和复用线程的技术，可以有效提高应用程序的性能和响应速度。线程池的主要优点包括：

  • 减少线程创建开销：线程的创建和销毁是相对昂贵的操作，线程池可以复用已存在的线程，减少这些开销。
  • 控制资源使用：通过限制线程的数量，可以防止系统资源被过度消耗。
  • 提高响应速度：任务可以直接分配给已有线程，减少了等待时间。
  • 便于管理：可以更方便地管理线程的生命周期和状态。

# 工厂类创建

Java 中支持通过 java.util.concurrent.Executors 工厂类来创建，创建方法包括：

  • 固定大小线程池 (FixedThreadPool)： Executors.newFixedThreadPool(int nThreads)
    • 描述：创建一个固定大小的线程池。
    • 特点：线程池中的线程数量固定，如果所有线程都在执行任务，新提交的任务会被放入队列中等待。
    • 适用场景：适合处理大量耗时较短的任务，确保资源不会被过度消耗。
  • 单线程线程池 (SingleThreadExecutor)： Executors.newSingleThreadExecutor()
    • 描述：创建一个单线程的线程池。
    • 特点：只有一个工作线程，确保所有任务按顺序执行。
    • 适用场景：适合需要保证任务顺序执行的场景。
  • 可缓存线程池 (CachedThreadPool)： Executors.newCachedThreadPool()
    • 描述：创建一个可缓存的线程池。
    • 特点：线程池会根据需要创建新的线程，但会复用之前构造的线程。适用于执行大量短期异步任务的程序。
    • 适用场景：适合处理大量短生命周期的任务。
  • 定时任务线程池 (ScheduledThreadPool)： Executors.newScheduledThreadPool(int corePoolSize)
    • 描述：创建一个支持定时和周期性任务执行的线程池。
    • 特点：支持延迟执行和定期执行任务。
    • 适用场景：适合需要定时或周期性执行任务的场景。
  • 工作窃取线程池 (WorkStealingPool)： Executors.newWorkStealingPool(int parallelism) （Java 8 及以上版本）
    • 描述：创建一个工作窃取线程池。
    • 特点：使用 ForkJoinPool 实现，线程之间可以 “窃取” 任务，提高并行度和效率。
    • 适用场景：适合处理大量细粒度的任务，提高并行处理能力。

# 核心参数

通过 new 方法创建的线程池实例具备以下的方法参数:

  1. corePoolSize：
    • 含义：线程池的基本大小，即即使空闲也会一直保留的线程数。
    • 作用：当新任务提交时，如果当前线程数小于 corePoolSize，即使有空闲线程，也会创建新线程来处理任务。
  2. maximumPoolSize：
    • 含义：线程池允许的最大线程数。
    • 作用：当线程池中的线程数达到 corePoolSize 后，如果继续提交任务且任务队列已满，则会创建新线程，直到线程数达到 maximumPoolSize。
  3. keepAliveTime：
    • 含义：线程池中超过 corePoolSize 的空闲线程等待新任务的最长时间。
    • 作用：如果线程池中的线程数大于 corePoolSize，并且某个线程空闲时间超过了 keepAliveTime，则该线程会被终止。
  4. unit：
    • 含义：keepAliveTime 的时间单位，如 TimeUnit.SECONDS、TimeUnit.MINUTES 等。
    • 作用：指定 keepAliveTime 的时间单位。
  5. workQueue：
    • 含义：用于保存等待执行任务的阻塞队列。
    • 常见类型：
      • ArrayBlockingQueue：基于数组的有界阻塞队列。
      • LinkedBlockingQueue：基于链表的无界阻塞队列。
      • SynchronousQueue：不存储元素的阻塞队列，每个插入操作必须等待一个移除操作，反之亦然。
      • PriorityBlockingQueue：具有优先级的无界阻塞队列。
      • DelayQueue：延迟任务队列，元素为 Delayed 对象，只有延迟期满的元素才能从队列中 take 出。
  6. threadFactory：
    • 含义：用于创建新线程的工厂。
    • 作用：可以自定义线程的名称、优先级等属性。
  7. handler：
    • 含义：拒绝策略，当线程池和任务队列都已满时，用于处理新提交任务的策略。
    • 常见策略：
      • AbortPolicy：默认策略，直接抛出 RejectedExecutionException 异常。
      • CallerRunsPolicy：由调用线程（提交任务的线程）执行该任务。
      • DiscardPolicy：直接丢弃任务，不抛出异常。
      • DiscardOldestPolicy：丢弃队列中最老的任务，然后重新提交新任务。
    • 除了 JDK 默认提供的四种拒绝策略，我们可以通过实现 RejectedExecutionHandler 接口根据自己的业务需求去自定义拒绝策略。

# 线程池的工作流程

当创建一个线程池后，不断赋予线程池任务，线程池会根据以下流程来处理任务：

  1. 检查当前线程数是否小于 corePoolSize：
    • 如果当前线程数小于 corePoolSize，线程池会创建新线程来处理新任务，即使其他线程是空闲的。
  2. 将任务添加到任务队列：
    • 如果当前线程数已经达到 corePoolSize，线程池会尝试将新任务添加到任务队列中。
    • 任务队列的选择会影响线程池的行为。
  3. 检查当前线程数是否小于 maximumPoolSize：
    • 如果任务队列已满且当前线程数小于 maximumPoolSize，线程池会创建新线程来处理新任务，直到线程数达到 maximumPoolSize。
  4. 执行拒绝策略：
    • 如果当前线程数已经达到 maximumPoolSize 且任务队列已满，线程池会根据配置的拒绝策略处理新任务。
    • 常见的拒绝策略包括：
      • AbortPolicy：默认策略，直接抛出 RejectedExecutionException 异常。
      • CallerRunsPolicy：由调用线程（提交任务的线程）执行该任务。
      • DiscardPolicy：直接丢弃任务，不抛出异常。
      • DiscardOldestPolicy：丢弃队列中最老的任务，然后重新提交新任务。

# 开启任务

在获取到线程池实例后，可以通过以下方式开启任务：

  • 使用 ExecutorService 的 submit 方法可以返回一个 Future 对象，用于获取任务的执行结果或取消任务。
  • 使用 ExecutorService 的 execute 方法没有返回值，适用于不需要获取任务结果的场景。

任务执行流程

  1. 线程获取任务：
    • 线程池中的空闲线程会从任务队列中获取任务。
    • 获取任务的方式取决于任务队列的类型。例如，LinkedBlockingQueue 使用 take 方法阻塞地获取任务，而 SynchronousQueue 使用
      poll 方法非阻塞地获取任务。
  2. 执行任务：
    • 线程获取到任务后，会调用任务的 run 方法来执行任务。
    • 任务执行过程中，如果任务抛出异常，线程池会捕获异常并处理。
  3. 任务完成：
    • 任务执行完成后，线程会返回任务队列，等待下一个任务。
    • 如果任务是通过 submit 方法提交的，Future 对象的状态会被更新，调用者可以通过 get 方法获取任务的结果。
